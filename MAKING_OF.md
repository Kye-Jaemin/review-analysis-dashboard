# 생성형 AI로 서비스를 만들다 — 제작 과정 이야기

### 소비자 리뷰 분석 Agent — 만들어진 과정 (Making-of)

> **이 문서의 목적**
> "생성형 AI를 실제 업무에 쓰면 무엇이, 어떻게 달라지는가"를
> 코드 한 줄 없이 **개념·사례·교훈** 중심으로 전달합니다.
> 대상: 비개발 실무자 · 리더 · 의사결정자.

🔗 **데모**: https://review-dashboard-54y4.onrender.com/

## 한 장 요약 (Executive Summary)

- **무엇을 만들었나** — 여러 채널(앱스토어·구글플레이·레딧·웹)의 소비자 리뷰를 자동으로 모아, 주제별로 분류하고, 감정을 점수화해 한눈에 보여주는 웹 서비스.
- **어떻게 만들었나** — 기획부터 코딩·디자인·버그수정·배포까지 **생성형 AI(클로드)를 페어 파트너로** 두고, 사람이 방향을 잡고 검증하는 방식으로.
- **얼마나 걸렸나** — **실작업 9일 / 3주**에 약 **2.3만 줄** 규모의 풀스택 서비스를 1인이 완성.
- **핵심 메시지** — 생성형 AI는 "버튼 한 번에 완성품"이 아니라, **방향을 주는 사람과 빠르게 반복하는 도구**다. 생산성은 곧 **반복 속도 × 검증 습관**에서 나온다.

## 1. 가장 중요한 구분 — "AI 활용"에는 두 층이 있다

발표에서 이 둘을 섞으면 청중이 혼란스러워집니다. 먼저 분리하세요.

| | ① **AI로 만들었다** (도구) | ② **AI를 담았다** (기능) |
|---|---|---|
| 의미 | 기획서·코드·디자인을 AI가 작성 | 완성된 앱이 사용자를 위해 AI를 실행 |
| 비유 | AI라는 **숙련공과 함께 집을 지음** | 집 안에 **똑똑한 가전을 설치** |
| 이 프로젝트의 예 | 3주 만에 서비스 전체를 구축 | 리뷰를 자동 분류·감정 점수화 |

> 이 자료의 주인공은 **①(만든 과정)** 입니다.
> ②는 "AI 기능을 잘 설계하면 어떤 모습인가"를 보여주는 **보너스 사례**로 다룹니다.

## 2. 제작 과정 7단계 — 사람과 AI의 역할 분담

각 단계에서 **사람이 한 일**과 **AI가 한 일**을 나란히 보면, 협업의 그림이 분명해집니다.

| 단계 | 사람이 한 일 (방향·판단) | AI가 한 일 (생성·실행) |
|---|---|---|
| **1. 기획** | "무엇을, 왜 만들지"를 결정 | 상세 기획서 초안 작성 |
| **2. 뼈대 세우기** | 기술 방향 승인 | 폴더 구조·기본 틀 생성 |
| **3. 기능 개발** | 원하는 기능을 말로 요청 | 동작하는 화면·로직 구현 |
| **4. 디자인 다듬기** | "더 고급스럽게" 같은 피드백 | 애니메이션·시각효과 적용 |
| **5. 버그 수정** | 문제 상황 전달 | 원인 찾고 수정 |
| **6. 품질 점검** | 합격 기준 설정 | 자동 테스트 작성·실행 |
| **7. 배포** | 공개 결정 | 배포 설정·실서버 반영 |

**핵심**: AI는 "어떻게(How)"를 빠르게 만들고, 사람은 "무엇을·왜(What/Why)"와 **"이게 맞나(검증)"**를 책임집니다.

---

## 3. 이 프로젝트가 보여주는 일하는 방식 — "작게, 자주, 검증하며"

실제 개발 기록(커밋 50개)은 그 자체로 좋은 교재입니다.

- **한 번에 완성이 아니라 50번의 작은 개선** — "리뷰 모달 위치 수정", "차트 색상 정리", "이름 변경"처럼 잘게 쪼갠 단위로 전진.
- **왜 작게?** — 작게 만들면 **틀렸을 때 빨리 되돌리고, 검증도 쉽기** 때문. AI가 만든 큰 덩어리를 통째로 믿는 것은 위험.
- **버그도 대화로 해결** — 에러 메시지를 그대로 AI에게 보여주면 원인과 수정안을 제시. 사람은 "정말 고쳐졌는지"만 확인.

> 💡 **업무 적용 교훈**
> AI 협업의 생산성은 *"한 번에 완벽한 지시"*가 아니라
> *"빠른 요청 → 결과 확인 → 다음 요청"의 회전 속도*에서 나옵니다.

## 4. 보너스 — "AI 기능을 똑똑하게 설계한다"는 것 (②층)

앱 안에서 AI가 리뷰를 분석할 때, 그냥 "AI에게 맡김"이 아니라 **신뢰할 수 있게 길들이는 장치**를 넣었습니다. 비개발자에게도 중요한 개념들입니다.

| 설계 장치 | 쉬운 설명 | 왜 중요한가 |
|---|---|---|
| **같은 입력 → 같은 결과** | AI의 "기분 따라 달라짐"을 끔 | 보고서 숫자가 매번 바뀌면 신뢰 불가 |
| **예시 먼저 보여주기** | 기준이 되는 분류 예시를 함께 제공 | 결과가 들쭉날쭉하지 않고 일관되게 수렴 |
| **3중 안전장치** | AI가 규칙 어기면 시스템이 걸러냄 | "장점인데 단점이 섞이는" 오류 방지 |
| **반복 호출 캐싱** | 같은 작업은 다시 안 시킴 | **AI 사용 비용 절감** |
| **숫자 보정** | AI 추정치를 실제 데이터로 맞춤 | 차트와 실제 건수 불일치 방지 |
| **작업별 모델 선택** | 가벼운 일엔 저렴한 AI, 중요한 일엔 고급 AI | 비용 ↔ 정확도 균형 |

> 💡 **교훈**: 생성형 AI를 제품에 넣을 때 진짜 실력은
> **"환각(틀린 답)·비용·일관성"을 어떻게 통제하느냐**에서 갈립니다.

### 4-1. 더 자세히 — "작업별 모델 선택" 전략

같은 AI라도 **여러 등급(티어)**이 있습니다. 빠르고 싼 모델부터, 느리지만 똑똑한 모델까지. 이 서비스는 **세 등급을 작업 성격에 맞춰** 씁니다.

**인력 배치에 비유하면**

> - **Haiku (경량·고속·저비용)** = 일을 빠르게 쳐내는 **실무 담당자**
> - **Sonnet (균형)** = 속도와 정확도가 모두 좋은 **중간 관리자**
> - **Opus (최고 정확·고비용)** = 중요한 판단을 맡기는 **전문가**
>
> 모든 일을 전문가에게 시키면 비싸고 느립니다. **일상 업무는 실무자가, 핵심 판단만 전문가가** — 회사 운영과 똑같은 원리입니다.

**실제로 어떤 작업에 어떤 모델을 쓰는가**

| 작업 | 성격 | 기본 모델 | 왜 이 선택인가 |
|---|---|---|---|
| **리뷰 분류 + 감정 점수** (핵심·대량) | 수백~수천 건 반복 | **Haiku** *(사용자가 필요 시 Sonnet/Opus로 승급)* | 대량 처리라 속도·비용이 관건. **중요한 분석만 사람이 고급 모델로 올림** |
| **주제 자동 도출 (Top 10)** | 데이터 1회 요약 | 위 분석에서 고른 모델을 그대로 사용 | 분류 작업과 한 묶음으로 처리 |
| **감정 원인 마인드맵** | 보조 요약, 자주 재생성 | **Haiku** | 탐색용이라 가볍고 저렴하게 |
| **리뷰 번역** | 짧은 단건 변환 | **Haiku** | 가볍고 빈번한 작업 |
| **업체 강·약점 원인 분석** | 정밀 추출 | **Haiku** + *온도 0 (재현성)* | 비용보다 **"같은 결과 보장"**이 더 중요 |
| **경쟁력 심화 분류·기준 매핑** | 분류 1회 | **Haiku** + *온도 0 + 캐시* | 결정성·비용을 동시에 잡음 |

**핵심 원칙 세 가지**

1. **기본은 가장 가벼운 모델(Haiku)** — 대부분의 작업은 빠르고 저렴하게. *불필요한 고급 모델 사용 = 돈 낭비*.
2. **승급은 사람이, 핵심 작업에만** — 정말 중요한 인사이트가 필요한 메인 분석에서만 버튼 하나로 상위 모델 선택. (마지막 선택은 기억되어 재입력 불필요)
3. **재현성이 필요하면 "온도 0"** — 보고서 숫자가 매번 달라지면 안 되는 작업은 AI의 무작위성을 끄고, 같은 작업은 캐시로 재호출조차 막아 **비용까지 절감**.

> 💡 **업무 적용 교훈**
> "가장 좋은 AI를 쓰자"가 아니라 **"작업마다 맞는 등급을 쓰자"**가 정답입니다.
> 모델 선택은 곧 **비용 통제 전략**이며, 비개발 리더가 가장 먼저 챙겨야 할 의사결정 포인트입니다.

## 5. 정직한 한계 — AI가 "혼자" 못한 것

신뢰받는 발표는 한계도 말합니다.

- **방향과 취향은 사람 몫** — "어떤 기능이 필요한가", "이 화면이 임원에게 와닿는가"는 AI가 정해주지 않음.
- **도메인 판단은 사람이** — 예: 같은 회사가 스토어마다 다른 이름으로 등록된 경우를 합치는 규칙은 사람이 정의.
- **검증 없이는 위험** — AI 결과물은 *반드시* 테스트·사람 검토를 거쳐야 함. 그래서 자동 테스트 19종을 함께 운영.
- **보안·비용은 설계 사항** — 무단 사용을 막는 접근 장치, 비용 통제는 사람이 의도적으로 넣음.

## 7. 발표용 한 줄 결론 (Take-aways)

1. **AI는 "완성 버튼"이 아니라 "빠른 파트너"** — 방향은 사람이, 속도는 AI가.
2. **생산성 = 반복 속도 × 검증 습관** — 작게 자주, 그리고 항상 확인.
3. **AI 기능의 진짜 실력은 통제력** — 일관성·비용·환각을 다루는 설계.
4. **결과는 명확** — 1인·3주로 풀스택 서비스. 우리 업무에도 같은 레버리지가 가능하다.
